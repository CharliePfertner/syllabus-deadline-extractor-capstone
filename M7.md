## Data model changes

### Option A — Full-text search (recommended for large datasets)

Add a `search_vector tsvector` column to the `notes` table, populated via a trigger on insert and update. Weight the `title` column higher than `body` (A vs B) so title matches rank first. Add a `GIN` index on the new column for fast lookups.

### Option B — Trigram search (simpler, fine for < 10k notes)

No schema change required. Enable the `pg_trgm` extension and add `GIN` trigram indexes on `title` and `body`. Query with `ILIKE`. Swap for Option A when performance degrades.

---

## Route changes

### Extend the existing list route

```
GET /notes?q={query}&page={n}&limit={n}
```

| Parameter | Default | Notes |
|-----------|---------|-------|
| `q`       | —       | URL-encoded search string. Absent = return all notes (existing behaviour). |
| `page`    | `1`     | 1-based page number. |
| `limit`   | `20`    | Max results per page; server cap at `100`. |

No new endpoint needed — extending the existing list route keeps the API surface small and is fully backward-compatible.

### Optional: autocomplete

```
GET /notes/search/suggestions?q={prefix}
```

Returns up to 5 title suggestions for the given prefix.

### Response shape additions

Add two new fields to each note object in search results:

| Field        | Type                              | Description |
|--------------|-----------------------------------|-------------|
| `matched_in` | `"title" \| "body" \| "both"`   | Where the match was found. |
| `snippet`    | string                            | ≤ 160 char excerpt centred on the first match, with match position included. |

Also add `total_count` and `query` at the response envelope level.

### Loading state

Show skeleton rows while the request is in-flight. Match skeleton height to the result row height to avoid layout shift.

### Empty state

When the query returns 0 results, show *No notes matching "{query}"* with a clear/reset button. Do not show the normal empty-notes onboarding prompt.

### Result row anatomy

Each result row contains three lines: the note title (with match highlighted), a body snippet (with match highlighted), and a metadata row showing last-modified date and word count.

## Edge cases to handle

### Input validation

| Case | Handling |
|------|----------|
| Empty / whitespace-only query | Treat as no search; revert to full note list. Never fire an API call for blank strings. |
| Very long queries | Cap client input at 200 chars; enforce server-side and return `400` with a message. |
| Special characters | Always use parameterised queries — never string-concatenate user input into SQL. |
| Unicode and emoji | Normalise to NFC before indexing and querying so accented variants match correctly. |

### UX and performance

| Case | Handling |
|------|----------|
| Stale in-flight responses | Cancel the previous request on each new keystroke to prevent out-of-order rendering. |
| Notes with no body | Skip the body snippet gracefully; show title match and metadata row only. |
| Very large result sets | Paginate at the API level; use scroll-triggered pagination or a "Load more" button in the UI. |
| Permissions | Search results must respect existing note visibility rules. Never leak private or archived notes through the search API. |
| Index lag on new notes | Newly created notes may not appear immediately if using async indexing. Synchronous trigger-based updates (Option A) avoid this. |
